# Motion design — current state (2026)

## The accessibility floor (binding)

`prefers-reduced-motion: reduce` is the user-system query that signals "I get sick / distracted by motion." Honoring it is not optional.

What MUST be disabled when the query matches:
- Scroll-triggered fade/slide reveals (show the final state immediately)
- Decorative loops (auto-rotate, marquees, animated illustrations)
- Number counters and progress reveals (jump to final value)
- Page transitions (Astro view transitions, if used)
- Parallax of any range
- `transform`-based entrance animations

What MAY stay even with reduced-motion:
- Color/background state transitions ≤200ms (still useful as feedback)
- Focus ring transitions
- `scroll-behavior: smooth` — but ALSO disable if you want strict; the safe pattern is `auto` under the query

Standard CSS pattern (already in `SiteLayout.astro`):

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

WCAG 2.2 SC 2.3.3 Animation from Interactions (AAA, but Pat LLC ships AA + considers AAA as a target): non-essential animation triggered by interaction MUST be able to be disabled. Reduced-motion is the canonical mechanism.

## Performance — animatable properties

Animate only `transform` and `opacity`. Both are GPU-composited; the browser doesn't recalc layout or paint. Animating `width`, `height`, `top`, `left`, `margin`, `padding` triggers layout (reflow) and is a 60fps killer on weak hardware.

| Property class | Cost | When OK |
|---|---|---|
| `transform`, `opacity` | Compositor only — cheap | Always |
| `color`, `background-color`, `border-color`, `box-shadow` | Paint — moderate | OK for short state transitions ≤300ms |
| `width`, `height`, `top`, `left`, `padding`, `margin` | Layout — expensive | NEVER for 60fps animation; use `transform: scale()` or `translate()` instead |
| `filter`, `backdrop-filter` | Paint, expensive on mobile | Sparingly; test on mid-tier mobile |

`will-change: transform` is a hint, not magic — add it the frame BEFORE animation starts (e.g. on hover-intent), not permanently. Permanently-applied `will-change` allocates GPU layers continuously and can hurt scroll perf.

## Frame budget

| Frame rate | Budget per frame | Use case |
|---|---|---|
| 60fps | 16.67ms | Standard target |
| 90fps | 11.11ms | High-refresh devices (most Pixel + iPhone Pro since 2022) |
| 120fps | 8.33ms | Newer iPad Pro, gaming displays |

Pat LLC budget: design for 60fps on mid-tier mobile (Moto G Power equivalent — Lighthouse default device). High-refresh devices get smoother for free; low-end devices need the discipline.

## Scroll-triggered animation — current best practice

`IntersectionObserver` is the right tool. Add the class once on intersection, then `unobserve` so it doesn't re-fire.

```js
const observer = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (entry.isIntersecting) {
      entry.target.classList.add('reveal--visible');
      observer.unobserve(entry.target);
    }
  }
}, { threshold: 0.15, rootMargin: '0px 0px -10% 0px' });

document.querySelectorAll('.reveal').forEach(el => observer.observe(el));
```

Anti-pattern: scroll listener (`window.addEventListener('scroll', ...)`) running calculation per scroll event. Even with `requestAnimationFrame` throttling, this loses to `IntersectionObserver` on perf and elegance.

**Scroll-driven animations API** (`@scroll-timeline`, `view-timeline`) is now stable in Chrome 115+ but Safari/Firefox lag. Use only with progressive enhancement — fallback must be the static (final) state, not broken.

## Anti-patterns currently shipped on bad sites

(see `motion-design` SKILL for the full list — this is the "what's especially current 2026 about avoiding")

- **Scroll-jacking on long pages** — full-bleed scroll-snap, replacing native scroll velocity with controlled segments. Hostile on touch devices, breaks reduced-motion users, breaks middle-click scroll, breaks skip-to-section.
- **AI-generated decorative motion** (Lottie / SVG animations generated by tools without restraint) — usually 200KB+, frame-rate poor, doesn't respect reduced-motion.
- **Auto-cycling video backgrounds** — even with prefers-reduced-motion, autoplay video carries bandwidth + battery cost.
- **Hover-only reveals** that don't have a touch equivalent — hostile on tablet/mobile.
- **Scroll-driven typography** (text size changes on scroll) — disorienting and reading-hostile.

## Eye-tracking + cognitive research relevant to motion

- **Banner blindness extends to animated banners** — Nielsen Norman, multiple studies. Auto-rotating hero banners get ignored.
- **Animation captures attention reliably; the cost is splitting attention from primary content.** Use motion to direct attention to the SINGLE next decision, not to decorate.
- **Reading interruption from peripheral motion is high.** Avoid animation playing while the user is reading body content (i.e. don't trigger reveals continuously as they scroll a paragraph).
